# iOS Translation

Two-prompt strategy for translating the Flutter/Android camera library to native iOS/Swift.

## Workflow

```
1. Run setup.sh          → Packs codebase with repomix, copies reference docs
2. Add screenshots       → Place app screenshots in screenshots/
3. Run Prompt 1          → Cartographer audits the codebase → writes to output/
4. Review output/        → Verify completeness, correct gaps
5. Run Prompt 2          → Architect designs iOS app → writes to design/
6. Review design/        → Adjust phases, resolve open questions
7. Begin implementation  → Use Phase 1 from design/02-implementation-phases.md
```

## Documentation Format

All component documentation uses the L1-L5 layered format:

- **L1: Purpose** — Why it exists, architectural role
- **L2: Contracts** — Behavioral invariants (must be replicated on iOS)
- **L3: Decisions** — Design tradeoffs with rationale and source
- **L4: Edge Cases** — Guards, watchdogs, timing values with tuning history
- **L5: Android Reference** — Implementation mechanism (for context, not translation)

## Development Phases (iOS)

1. Camera capture + controls + state machine + error handling
2. Metal processing pipeline (replace raw preview)
3. C++ sink integration (ObjC++ bridge)
4. Pipeline fan-out + performance tuning
5. Capture, recording, and feature parity
