# 08 — UI

Primary-owner file for the **SwiftUI layer**: view composition, preview surfaces, controls,
scenePhase wiring, orientation lock. Engine internals are in `01-system-shape.md`; the
cross-subsystem sequence for `.inactive` / `.background` is `02-concurrency.md` §Sequence A.

---

## View topology

Single SwiftUI root `CameraView` per ADR-01 extended. Two `MTKView`s wrapped in two
`UIViewRepresentable`s for split-screen preview.

```
CameraView (SwiftUI, @MainActor per ADR-21)
├── splitPreview           ── ZStack with natural + processed previews side-by-side
│   ├── NaturalPreview     ── UIViewRepresentable wrapping MTKView (natural drawable)
│   └── ProcessedPreview   ── UIViewRepresentable wrapping MTKView (processed drawable,
│                              rendered from shared IOSurface by async consumer lane)
├── bottomBar              ── Settings / Calibrate Color / Capture / Record / Resolution
├── expandedBar (conditional) — ISO / Shutter / Focus / Zoom controls
├── calibrationSidebar (conditional) — WB / BB / Brightness / Contrast / Saturation /
│                               Gamma / Reset
├── recordingIndicator (conditional) — elapsed timer + stop button
├── captureBanner (transient) — "Image saved: ..."
├── errorOverlay (fatal)   — blocking dialog
└── errorToast (non-fatal) — transient non-blocking banner
```

ViewModel (`@Observable`, `@MainActor`) holds:
- `sessionState: SessionState`, `recordingState: RecordingState`, `currentError: CameraError?`
- `liveFrameResult: FrameResult?`, `deviceSnapshot: DeviceStateSnapshot?`
- `deliveryStats: FrameDeliveryStats?` (debug builds)
- Slider values for `CameraSettings` and `ProcessingParameters` — mirrors persisted state
  on load; debounced before dispatch to engine.

`CameraView.body` uses `.task { await viewModel.bind(engine:) }` and `.task(id: scenePhase)`
per ADR-28 — auto-cancels on view disappear.

## UIViewRepresentable — MTKView wrappers

Per ADR-01 extended, each preview surface is a thin `UIViewRepresentable`:

```
NaturalPreview (UIViewRepresentable)
    makeUIView: creates MTKView; hands to engine via
                engine.bindNaturalPreview(mtkView:)
    updateUIView: no-op (drawable redraws on demand from CADisplayLink / engine commit)
```

Wrapping keeps the `MTKView` implementation detail inside `CameraView.swift` per ADR-01
§The three-layer sandwich anti-pattern. The view representable is ~30 lines; belongs with
the view.

- `colorPixelFormat`: `constants.md#WORKING_PIXEL_FORMAT` (`.rgba16Float`) when the natural
  preview renders to its drawable via a same-format blit (`04-metal-pipeline.md` Pass 3).
- `framebufferOnly = false` — we blit *to* the drawable; hints to Metal that read may occur.
- `isPaused = true`; `enableSetNeedsDisplay = false` — the engine drives the display by
  committing the per-frame command buffer on the delivery queue; no `CADisplayLink` from
  the view side.

The processed preview is an async consumer lane (per `04-metal-pipeline.md` §Command graph
note) — the `MTKView` renders from a shared IOSurface-backed texture that the engine writes
each frame; the render is asynchronous, not gated on the 30Hz clock. The async consumer
machinery is exactly the canny-preview shape in ADR-01 §Direct GPU outputs vs async
consumers.

## Preview surface rebind

Per `domain-revised/05-resource-lifecycle.md` §Preview Surface Rebind, on
`constants.md#PREVIEW_SURFACE_FAILURE_THRESHOLD` consecutive drawable acquisition failures,
`04-metal-pipeline.md` §Preview surface rebind emits `previewSurfaceLost`; the UI layer
re-creates the `MTKView` within the representable's `updateUIView` (triggered by a state
flip in the view model) and hands a new view to the engine. The capture session is
unaffected.

Implementation detail: the `UIViewRepresentable`'s `updateUIView` branch on a state flip;
no full SwiftUI rebuild is needed.

## scenePhase wiring

`CameraView` observes `.environment(\.scenePhase)` (SwiftUI) and runs:

```
.task(id: scenePhase) {
    await viewModel.handleScenePhase(scenePhase)
}
```

ViewModel's `handleScenePhase`:

| Phase | Action |
|---|---|
| `.active` | Set GPU gate = true via engine; no session restart triggered from UI (rely on system `interruptionEnded`). |
| `.inactive` | Set GPU gate = false via engine; await scheduled-edge via `lastCommittedCommandBuffer.waitUntilScheduled()` per ADR-09. |
| `.background` | Call `engine.backgroundSuspend()`; engine handles session stop + recording drain. |

The wiring matches `02-concurrency.md` §Sequence A. D-06's strict policy means `.inactive`
always gates — even for notification banners.

Orientation lock: landscape-right only. SwiftUI `.supportedInterfaceOrientations(.landscape)`
or the UIKit-level `UISupportedInterfaceOrientations` in Info.plist (implementation choice;
the domain requires landscape-right per domain 09 §Orientation).

## Bottom bar

Always present during an active session. Five items per domain 09 §Bottom Controls Bar:

| Control | Behavior | Enabled in state |
|---|---|---|
| Settings | Toggles expanded bar visibility | `.streaming` |
| Calibrate Color | Toggles calibration sidebar | `.streaming` |
| Capture | Triggers `engine.captureImage(outputPath: nil)` | `.streaming` |
| Record / Stop | Toggles `engine.startRecording(options:)` / `engine.stopRecording()` | `.streaming` (start) / `.streaming` & `isRecording` (stop) |
| Resolution display | Shows `activeCaptureResolution` as `WxH` text | any state except `.closed` |

Disabled states show the control greyed per `04-avfoundation.md` §scenePhase; the
`.opening` / `.recovering` / `.paused` / `.error` / `.closed` states follow domain 09 §State-
Driven UI Behavior (authoritative). Wired via `viewModel.sessionState`.

## Expanded bar

Collapsible drawer containing individual camera parameter controls when `.streaming`:

| Control | Display | Input | Dispatch |
|---|---|---|---|
| ISO | `AUTO` or integer (from `liveFrameResult.iso`) | Segmented toggle (auto ↔ manual) + slider | `engine.updateSettings(.init(isoMode:.manual, iso:...))` |
| Shutter | `AUTO` or `1/X` (from `liveFrameResult.exposureTimeNs`) | Same shape as ISO | `engine.updateSettings(.init(exposureMode:.manual, exposureTimeNs:...))` |
| Focus | `AUTO` or lensPosition text | Toggle + slider | `engine.updateSettings(.init(focusMode:.manual, focusDistance:...))` |
| Zoom | `Nx` (from `liveFrameResult` / hardware zoom observer) | Slider + pinch gesture | `engine.updateSettings(.init(zoomRatio:...))` (coalesced) |

### ISO/shutter coupling UX

When the user toggles ISO or shutter from auto to manual, the ViewModel always dispatches
both `isoMode` and `exposureMode` = `.manual` in the same `updateSettings` call, matching
the engine-side Rule 1/2 after-merge resolution. Displaying a "couples automatically"
affordance avoids user confusion; the engine-side coupling enforces correctness regardless
of the UX signal.

### Slider coalescing

High-Hz slider input is coalesced at 60 Hz via a debouncer in the ViewModel before
dispatch. Per `03-camera-session.md` §Configuration windows, holding `lockForConfiguration`
per-pixel is wasteful; the debouncer prevents a per-drag-pixel commit storm.

Implementation: the ViewModel owns a `Task` that reads slider updates from an
`AsyncStream.bufferingNewest(1)` (state-shape), sleeps 16 ms after each, and dispatches the
latest value to the engine. Slider changes during the sleep overwrite the buffered value.

## Color calibration sidebar

Controls per domain 09 §Color Calibration Sidebar:

| Control | Range (`constants.md`-free) | Dispatch |
|---|---|---|
| White Balance / Calibrate | — | `engine.sampleCenterPatch()` → compute gains → `engine.updateSettings(.init(wbMode:.manual, wbGainR:..., wbGainG:..., wbGainB:...))` |
| Black Balance / Calibrate | — | `engine.sampleCenterPatch()` → compute offsets → `engine.setProcessingParameters(.init(blackR:..., blackG:..., blackB:...))` |
| Brightness | `[-1.0, 1.0]` | `engine.setProcessingParameters(...)` (merge with current) |
| Contrast | `[0.0, 2.0]` | same |
| Saturation | `[-1.0, 1.0]` (API) displayed as `[0.0, 2.0]` (UI label) | same |
| Gamma | `(0.0, ...]` | same |
| Reset all | — | `engine.setProcessingParameters(.identity)` |

The calibration logic (gains and offsets from sampled R, G, B) lives in the ViewModel, not
the engine. The engine only applies `ProcessingParameters` — the math is UI logic per
`07-settings.md` §Calibration flows.

Sidebar does not block the preview; it is dismissible and remains usable during recording
per domain 09 §Recording Indicator.

## Recording indicator

Appears when `recordingState == .recording`:
- Record button transforms to Stop button.
- Red indicator showing elapsed time in `MM:SS` format, computed from
  `viewModel.recordingStartedAt` (captured at the `.recording` transition).
- Preview continues uninterrupted.
- Calibration sidebar remains usable.

Timer implementation: `TimelineView(.periodic(from: start, by: 1.0))` re-renders the
elapsed label once per second without requiring a per-frame update tied to the frame
clock.

---

## Capture banner

After a successful `captureImage` return, the ViewModel sets `captureBanner = .init(path:
output.filePath)` for a transient period (e.g. 3 seconds). Banner text: `"Image saved:
<path>"`. Banner dismisses on timer or user tap. No confirmation required.

Error: if capture fails (`StillCaptureError` surfaces as `EngineError.fatal(...)` or raw
`StillCaptureError`), the error toast fires instead.

## State-driven UI behavior

Per domain 09 §State-Driven UI Behavior (authoritative; see
`see domain-revised/09-ui-behaviors.md#state-driven-ui-behavior for the authoritative
statement`):

| State | UI |
|---|---|
| `.opening` | Loading spinner; all controls disabled. |
| `.streaming` | Full UI; preview alive; controls enabled. |
| `.recovering` | Non-blocking banner ("Recovering camera..."); preview frozen or last-frame; controls disabled. |
| `.paused` | Preview frozen; capture/record disabled; settings disabled (device may be in stale state). |
| `.error` (fatal) | Blocking error dialog; Retry (calls `engine.close()` then `engine.open(...)`) and Dismiss actions. |
| `.closed` | Return to initial (pre-open) UI. |

---

## Error display

Non-fatal errors (`FRAME_STALL` / `FPS_DEGRADED` / `AE_CONVERGENCE_TIMEOUT` /
`RECORDING_TRUNCATED` / `SETTINGS_CONFLICT`) surface as transient banners at the bottom.
Duration: 3–4 s. Auto-dismiss.

Fatal errors (`isFatal: true`) surface as blocking dialogs with:
- Error message (human-readable).
- Retry button → `engine.close()` then `engine.open(configuration: …)`.
- Dismiss → close the view, release the engine.

The engine does not retry fatal errors itself (domain 06 §Fatal Error Sequence). UI drives
retry.

## FrameResult display

`frameResultStream` updates `viewModel.liveFrameResult` at
`constants.md#FRAME_RESULT_HEARTBEAT_HZ`. The expanded bar reads the live values (not the
slider-committed values) so the display reflects what the sensor is actually doing per
`07-settings.md` §Frame-result heartbeat.

`focusDistance == nil` (actively scanning): the focus control shows a scanning animation
(spinner or animated ellipsis). See domain 09 §FrameResult Display.

---

## Debug surface (development builds only)

Debug builds expose a long-press gesture on the Resolution display that opens a
`FrameDeliveryStats` overlay showing per-lane counters from D-11. Production builds omit
this; the `FrameDeliveryStats` stream is still consumed but not displayed.

Log-level control per domain 09 §Log Level Control: a hidden setting in debug builds
toggles `os_log` verbosity. Not part of the production UI.
