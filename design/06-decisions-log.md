# 06 — Decisions Log

Every significant design choice with alternatives considered and reversibility assessment.

| # | Decision | Alternatives considered | Chosen because | Reversibility |
|---|---|---|---|---|
| D-01 | `CameraEngine` as Swift `actor` (not class + serial queue) | (A) class + `DispatchQueue` serial queue + explicit locks; (B) class + OSLock | Swift actor provides compiler-enforced isolation — domain Invariant 1 (camera state exclusively serialized) is a compile-time guarantee, not a runtime promise. Actor re-entrancy at `await` points is well-understood and controlled. Serial queue approach requires manual discipline; actors make violations impossible to express. | Low — changing to a queue-based model would require significant refactoring of all caller sites that use `await engine.method()`. Worth doing only if actor re-entrancy causes subtle state bugs. |
| D-02 | VTFrameProcessor: **not used**; custom Metal compute shaders chosen | (A) `VTFrameProcessor` (iOS 26 VideoToolbox API); (B) CoreImage filter chain | `VTFrameProcessor` supports system-defined effects (motion deblur, super-resolution) — it does not expose arbitrary per-channel color transform pipelines. The required 5-stage color pipeline (black balance → brightness → contrast → saturation → gamma with Rec.709 coefficients) cannot be expressed through `VTFrameProcessor`'s effect enumeration. CoreImage filters would work but add overhead (NSObject-based pipeline, implicit texture copies) and the filter graph is harder to instrument. Custom Metal compute kernels are fast, inspectable, and directly instrumentable. | Medium — if Apple adds custom color pipeline support to `VTFrameProcessor` in a future SDK, it would be worth revisiting. |
| D-03 | `AVAssetWriter` + `AVAssetWriterInputPixelBufferAdaptor` for video recording with true GPU-to-encoder zero-copy via IOSurface-backed `CVPixelBuffer` pool | (A) `AVCaptureMovieFileOutput`; (B) VideoToolbox `VTCompressionSession` directly; (C) `AVAssetWriter` with `MTLTexture.getBytes` CPU blit | `AVCaptureMovieFileOutput` cannot encode GPU-processed frames — it writes the sensor's capture output directly, bypassing the Metal pipeline. `VTCompressionSession` adds unnecessary complexity (manual sample buffer construction, manual atom/moov management). `AVAssetWriter` + `AVAssetWriterInputPixelBufferAdaptor` is the standard iOS path. Zero-copy is achieved by (1) creating the pixel buffer pool with `kCVPixelBufferIOSurfacePropertiesKey` and `kCVPixelBufferMetalCompatibilityKey: true`, (2) wrapping each dequeued `CVPixelBuffer` as a `MTLTexture` via `CVMetalTextureCache`, (3) blitting the processed output into the wrapped texture with `MTLBlitCommandEncoder` (GPU-local, stays in IOSurface memory), (4) appending the pixel buffer to the adaptor — VideoToolbox maps the same IOSurface for encoding. `MTLTexture.getBytes` is **explicitly forbidden** on this path — it would violate the domain zero-copy invariant. Full implementation pattern documented in `03-metal-pipeline.md §GPU-to-Encoder Path`. | Low — `AVAssetWriter` is the standard iOS high-quality recording path. The IOSurface zero-copy integration is non-negotiable; alternatives all copy. |
| D-04 | ObjC++ bridge for OpenCV; direct Swift-C++ interop for `IFrameConsumer` | (A) Pure direct Swift-C++ interop for all C++ code; (B) Pure ObjC++ wrapper for all C++ code | OpenCV headers use templates, macros, and C++ exceptions that Swift's Clang module importer cannot bridge. `IFrameConsumer` is a simple non-template C++ interface that Swift can import directly. Two-layer approach gives the cleanest public API: Swift sees the generic interface directly (type-safe, no objc overhead), and OpenCV-specific code is safely encapsulated in `.mm` files. | Medium — if Swift-C++ interop improves to handle complex templates in a future Swift release, the ObjC++ layer could be removed. |
| D-05 | OpenCV distributed as **xcframework** (pre-built) | (A) CocoaPods; (B) SPM; (C) Build from source | Pre-built xcframework from opencv.org is official, signed, and avoids build-time toolchain complexity. CocoaPods is deprecated-trajectory. SPM has no official OpenCV package. Building from source requires maintaining CMake configuration across Xcode/SDK updates. | Medium — switching to SPM (if an official package appears) or source build (for binary size optimization) is possible without changing the bridging architecture. |
| D-06 | Edge detection result type: **contour list** (not binary pixel mask) | (A) Binary edge mask (`CVPixelBuffer`); (B) Edge coordinate list per-row | Contour list (`[EdgeContour]`) is compact (`O(edges)` not `O(pixels)`), natively `Sendable` as a Swift struct, and renders efficiently in SwiftUI `Canvas` via `Path`. Binary mask at full resolution is ~12.5 MB per frame — expensive to transfer from C++ consumer to `@MainActor` as a `Sendable` value. Per-row coordinate list loses shape information that `cv::findContours` provides for free. | Low — if dense mask is needed (e.g., for ML training data export), the consumer can be extended to optionally produce both. |
| D-07 | Split-screen preview: two separate `MTKView` instances | (A) Single `MTKView` with side-by-side blit; (B) `MTKView` + `UIImageView` | Two `MTKView` instances map cleanly to the two SwiftUI `UIViewRepresentable` wrappers. Each renderer is independently updateable. Single `MTKView` with two-region blit adds coordinate math complexity. `UIImageView` requires a CPU-side pixel buffer transfer for the natural stream, which is unnecessary. | Low — architectural; both views share the same Metal device, so combining is possible later. |
| D-08 | App lifecycle "fully invisible" → `scenePhase == .background` (not `.inactive`) | (A) `scenePhase == .inactive`; (B) UIApplicationDelegate `applicationWillResignActive` | Domain spec explicitly requires "fully invisible" trigger to avoid releasing camera on partial occlusion (dialog overlay, notification banner). SwiftUI `scenePhase == .inactive` fires on partial occlusion. `.background` fires only when the app is fully off-screen — exact match for domain intent. This is a direct resolution of U-07. | Low — this is the correct behavior per the domain spec; no reason to revisit. |
| D-09 | Settings persistence: **UserDefaults + Codable** | (A) Core Data; (B) SQLite; (C) Keychain | `CameraSettings` and `ProcessingParameters` are small Codable structs. `UserDefaults` is zero-setup, supports `Double` natively (no bit-packing workaround needed), and is appropriate for user-facing preferences. Core Data and SQLite are over-engineered for this use case. Keychain is for secrets. | Low — if settings grow significantly, migration to Core Data is straightforward. |
| D-10 | `ConsumerRegistry` as a separate Swift `actor` from `CameraEngine` | (A) Consumer registry inside `CameraEngine` actor | Separating the registry from the engine allows consumers to be registered/unregistered without holding the engine actor. It also enables the consumer dispatch path to be non-blocking relative to other engine operations. The registry's `dispatch(frame:to:)` is called from `CameraEngine` (actor-to-actor), which is the correct direction. | Medium — merging back into `CameraEngine` is possible if actor isolation between them causes performance issues from too many `await` hops. |
| D-11 | Camera orientation: use `AVCaptureConnection.videoRotationAngle` | (A) Manual UV transform matrix in Metal shader (as Android source did) | iOS AVFoundation handles sensor orientation via `AVCaptureConnection.videoRotationAngle`. Setting the correct rotation at session configuration time means the `CVPixelBuffer` arrives pre-rotated — no Metal shader UV matrix needed. This is simpler, more correct, and doesn't require re-determining the rotation angle per-frame. | Low — if AVFoundation rotation produces incorrect results for the target hardware, the Metal approach can be restored. |
| D-12 | `AsyncStream<T>` for state, error, frameResult delivery to ViewModel | (A) Combine `PassthroughSubject`; (B) Direct `@MainActor` method calls from engine | Swift concurrency `AsyncStream` is the idiomatic iOS 26 / Swift 6 approach. No Combine dependency. `for await` loops in `.task { }` modifiers auto-cancel on view disappear. Direct `@MainActor` calls from `CameraEngine` would create stronger coupling between engine and UI layer. | Medium — replacing with Combine is possible but adds a framework dependency; stick with AsyncStream. |
| D-13 | Focus distance: iOS `lensPosition` (0–1 normalized); **diopter display approximated** | (A) Attempt hardware-specific diopter-to-lensPosition calibration curve; (B) Expose raw `lensPosition` without diopter framing | iOS `AVCaptureDevice` uses `lensPosition` (0.0 = near, 1.0 = far), not diopters. Exact diopter conversion requires factory calibration data not available at design time. Design exposes `lensPosition` in manual focus control; the UI labels it as a relative focus control rather than diopters. The domain API contract preserves the `focusDistanceDiopters` field name for cross-platform compatibility, but the iOS value is the normalized position. This is documented as a known deviation in `07-ios-specific-risks.md`. | Medium — if hardware calibration data becomes available, the conversion function can be added without API change. |
| D-14 | SDR only — `BGRA8Unorm` throughout | (A) `rgba16Float` for HDR processing | The domain color pipeline is SDR-range (brightness `[-1,1]`, contrast `[0,2]`). All consumers receive `RGBA8888` per domain spec. SDR is correct for the current feature set. HDR is a forward-compatible upgrade path (change one line in the compute pipeline descriptor). | Low reversibility required — this is the correct decision now; HDR upgrade is additive. |
| D-15 | `sampleCenterPatch()` reads from Metal readback buffer (CPU-side) | (A) Dispatch a separate Metal compute pass to sample the patch; (B) Read from `MTKView.currentDrawable` | The readback buffer is already a CPU-accessible RGBA buffer. Reading a 96×96 window from it is trivial and runs on the actor — no additional GPU work needed. The histogram trimmed mean computation in Swift/CPU on 9,216 pixels takes < 1ms. Separate GPU pass adds command buffer overhead for no benefit at this size. | Low — if 96×96 proves insufficient, the patch size is a single constant change. |
