# 06 — Decisions Log

Product-specific decisions, each citing the ADR it follows or explicitly deviates
from. Alternatives listed; "follow ADR-## as written" is always included for any
deviation.

| D-## | Decision | Alternatives considered | Chosen because | Reversibility |
|---|---|---|---|---|
| **D-01** | `actor CameraEngine` is the single heavy isolation domain. State (state machine, retry counter, persisted settings, thermal state, consumer registry) lives on the actor. `AVCaptureSession` work hops to a dedicated serial `sessionQueue`; per-frame path runs on `deliveryQueue`. | (a) Plain `final class` on `sessionQueue` serial queue — "no actor", classical AVFoundation pattern; (b) Two actors (`CameraEngine` + `@globalActor ML`) — the rejected "sandwich theater"; (c) Follow ADR-02 as written. | Follows **ADR-02** and ADR-07. Actors give compile-time enforcement of domain invariant 1 (camera state exclusively serialized) without manual mutex discipline. The per-frame path correctly **does not hop** the actor (ADR-02 §Forbidden Task hop) — actor is touched only for non-per-frame ops. One-actor-per-lifecycle: `CameraEngine` has one lifecycle (open → stream → close). | Reversible: collapsing to a plain class on `sessionQueue` is mechanical (swap `actor` for `final class`, guard state with a `DispatchQueue.sync`). Don't unless the Sendable story becomes intractable — unlikely. |
| **D-02** | GPU transforms are **custom Metal compute/fragment shaders**, not `VTFrameProcessor`. | (a) `VTFrameProcessor` for all transforms; (b) Mix: `VTFrameProcessor` for downsample, custom for color ops; (c) Follow ADR-05 / G-21 as written. | Follows **ADR-05** + G-21. `VTFrameProcessor` exposes only system-defined effects (deblur, super-res, noise reduction, frame interpolation) — it cannot run the per-channel color chain from domain §03 or the RGBA16F→NV12 encoder compute. Verdict confirmed against WWDC25 Video Toolbox sessions. | Re-evaluate only if Apple adds programmable color pipeline support; rewriting a single shader is cheap. |
| **D-03** | Swift ↔ C++ via `.interoperabilityMode(.Cxx)` + `cxxLanguageStandard: .cxx20`. Zero `.mm` files. Public C++ headers contain only POD + `enum class` + `SWIFT_SHARED_REFERENCE` + C-ABI typedefs. | (a) Objective-C++ (`.mm`) bridging wrappers around OpenCV — the Swift 5-era pattern; (b) Pure C wrapper layer (`extern "C"` thin facade + Swift imports C only) — coarser but simpler; (c) Follow ADR-11 / ADR-12 as written. | Follows **ADR-11** and **ADR-12**. `.mm` files are unnecessary under Swift 6 C++ interop for common cases; maintaining an ObjC++ layer would be pattern-theater. OpenCV headers stay in `.cpp` files; Swift never imports them. `.interoperabilityMode(.Cxx)` is constrained to the `Cpp` and `InteropFacade` targets only (ADR-13). | Reversible: if a specific header defeats the importer, a C-wrapper fallback for that one type is isolated. |
| **D-04** | OpenCV is distributed as a pre-built `.xcframework` (iOS device arm64 + simulator slices) referenced via SPM `.binaryTarget(name: "opencv2", url:, checksum:)`. | (a) CocoaPods `pod 'OpenCV'`; (b) Building OpenCV from source via CMake External Project (Android-style); (c) SPM source build with a vendored submodule. | Speed and reproducibility. A pinned `.xcframework` checksum gives byte-reproducible builds and no Ruby toolchain. CocoaPods would add a workspace; source builds take a week to bring up and offer no benefit for a POC consumer. | Fully reversible: swap the `.binaryTarget` URL for a new release; C++ source doesn't change. |
| **D-05** | **Sendable strategy for frame buffers:** `CVPixelBuffer` refs cross into the C++ consumer pool wrapped inside `FrameSet` (`Sendable` by ADR-18 — it's a struct of IOSurface handles and POD metadata). The raw `CVPixelBuffer` type is `@unchecked Sendable` only where unavoidable (Apple has not marked it as of iOS 26 — G-13); elsewhere, `sending` parameter annotation (SE-0430) is used. | (a) Wrap every `CVPixelBuffer` in a `@unchecked Sendable` Swift box; (b) Copy pixels into a fresh `Data` on every handoff (defeats zero-copy); (c) Follow ADR-10 / ADR-18 as written. | Follows **ADR-10** + **ADR-18**. The IOSurface under a `CVPixelBuffer` is cross-process, cross-API coherent memory on Apple Silicon — designed for transfer. ARC keeps it alive across handoffs. `sending` enforces single-ownership at compile time. Copying defeats zero-copy and breaks the domain "single memcpy" invariant. | Low-reversibility: the whole pipeline depends on zero-copy; reverting to memcpy is a rewrite. |
| **D-06** | Edge-detection compositing source is **`FrameSet.processed`**, not `FrameSet.natural`. | (a) Natural (unprocessed) base — matches "raw + edges" analyst expectation; (b) User-selectable; (c) No deviation — domain is silent. | Domain §01 invariant 1: preview display is the processed output. Showing Canny on the natural image would diverge from the user's mental model and from the still-capture contract ("the user gets what they see" — domain §08). User-selectable is feature creep for a POC consumer. | Fully reversible: change one subscription call in `EdgeDetectionConsumer::configure` + one uniform in the composite kernel. |
| **D-07** | **C++ consumer result type for rendering: shared IOSurface-backed `MTLTexture`**, NOT a Sendable result struct. C++ writes composited pixels into a pre-allocated shared texture; Swift triggers mipmap blit + render. | (a) Sendable result struct `EdgeResult { bytes: [UInt8], w, h, bpr }` sent to `@MainActor` — copies N MB per frame; (b) `CVPixelBuffer` round-trip back through an `AsyncStream<CVPixelBuffer>` — adds an async hop to the render loop; (c) Follow ADR-13 as written. | Follows **ADR-13**. The rendering consumer returns **full-res RGBA**; copying that into a `Data` for every frame would allocate ~15 MB per frame. One shared IOSurface-backed texture (allocated **once** at engine setup per design/04 §6) is the correct primitive. Data consumers (future tracker, Sendable POD) still use Sendable structs — this D-07 applies only to rendering consumers. | Fully reversible: switch to a Sendable result struct if copy cost becomes acceptable (it won't at full-res 30 fps). |
| **D-08** | `trackerTex` storage mode is **always `.shared` (IOSurface-backed)** from engine setup. `naturalTex` and `processedTex` default to `.private` and flip to `.shared` on first subscriber attach (ADR-20). | (a) All three always `.shared` — simpler but wastes ~30% DRAM bandwidth when no consumer is attached; (b) All three dynamic (flip per subscribe) — cleanest but adds a rotation for a stream that will always have a subscriber in this product; (c) Follow ADR-20 as written. | Follows **ADR-20** and **G-25** and G-26. The edge-detection consumer is designed-in; `trackerTex` always has a subscriber (the POC) in the product. Making it permanently `.shared` eliminates the storage-mode flip for the tracker lane and simplifies pool sizing. Natural and processed remain dynamic because a user may turn off the processed-stream consumer. | Reversible per stream: set `.shared` at allocation; no API changes. |
| **D-09** | `TARGET_BITRATE_MBPS` is exposed as the `bitrate` API parameter on `startRecording`; a conservative default (e.g. 50 Mbps for 1600×1200 HEVC) is used until Phase 5 tuning. | (a) Hardcode 50 Mbps; (b) Use AVFoundation's `AVVideoQualityKey = .high`; (c) Follow domain §08 "Configurable? Yes" literal spec. | Domain §10 makes `bitrate` a caller-provided `Int?` on `startRecording`. Matching the API contract exactly is lower-cost than introducing a quality enum. Phase 5 tunes the default empirically on target hardware. | Fully reversible: change the default constant. |

**Deviation check.** D-06 and D-07 are the only entries that are product-specific
decisions not directly dictated by an ADR. Neither contradicts an ADR:

- D-06 is compatible with ADR-13 (any consumer source is permitted; domain/UX
  considerations pick processed).
- D-07 follows ADR-13's §"What never crosses the consumer boundary" — rendering
  via a shared IOSurface-backed texture is the explicitly named correct shape for
  large-pixel outputs.
